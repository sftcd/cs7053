What’s XML?
  • eXtensible Markup Language
    • A format for describing structured data
    • Current “flavor of the month” representation
    • Lots of B2B activity (e.g. SOAP)
  • Aims to provide separation between content and the representation of that content
    • e.g. XML nodes + stylesheet (also XML) -> HTML

DTDs and Schemas
  • Used to describe the structure of a set of XML documents
  • DTD = Document type definition
  • Schema
    • Basically more up-to-date and flexible than DTDs
  • XMLDSIG specification contains both
    • Schemas are definitive

The need to sign
  • XML is being used to describe:
    • Data in transactions & protocols
    • Document formats
    • Stored data formats
  • In various circumstances all of these may require:
    • Data origin authentication
    • Support for evidence maintenance
  • This is just the same as previously seen in other contexts, e.g. SMTP (S/MIME), EDIFACT (USC)
10
Requirements
  • Included some design “principles”
    • Represent XML signatures in XML
    • Need to be able to sign XML documents and other content
    • Signing implies a need for a canonical form
      • c14n = canonicalization (“c”… 14 letters... “n”)
    • No “trust” semantics
  • Caution: “sign” here includes symmetric mechanisms
    • So use of HMAC is also defined
11
More requirements
  • Be able to sign part or all of  an XML document
  • Be able to include the signature within a signed XML document
  • Be extensible
    • Multiple signers, signatures over signatures, algorithm independent…
  • Respect the “web”
    • Content may move, but signature should still be verifiable (if you can find the original content)
12
Non-requirements
  • It wasn’t part of this work to:
    • Deal with PKI issues
      • Specification supports (but doesn’t mandate) X.509 PKI, PGP or yet another PKI
    • Deal with the “meaning” of signatures
      • So, no distinction between authentication and non-repudiation here (phew!)
    • Deal with encryption (confidentiality)
      • See later though
13
Working method
  • Basically started with CMS
    • CMS (RFC2630) is the IETF version of pkcs#7 which is used for S/MIME
    • Another good idea!
  • Then
    • Translate to XML and make it better
    • Iterated on some XML W3C co-ordination issues
      • Esp. c14n
    • Tweak specification based on implementations

How do I sign?
  • Pick your algorithms...
  • You take the content…
  • Decide which bits you want to sign…
  • Digest it (them) using a hash algorithm…
  • Form a SignedInfo…
  • C14N it...
  • Apply your private (or secret) key…
  • Form up an XML <Signature>…
  • Maybe put the signature back in the document...

Some useful things…
  • Algorithms etc are represented as URIs:
    • e.g. “http://www.w3.org/2000/09/xmldsig#sha1”
  • Algorithm URIs used for:
    • Signing algorithms (rsa-sha1)
    • Digest algorithms (sha1)
    • C14N algorithms
      • “http://www.w3.org/TR/2000/WD-xml-c14n-20000710”
    • “Transforms” - how to select part(s) of a document

More about SignedInfo...
  • Basically determines the content that’s been signed
  • Signatures can be:
    • Enveloping
      • Signature “surrounds” signed content
    • Enveloped
      • Signature “surrounded” by signed content
    • Detached
      • Signed content is “elsewhere”
    • Once you understand these…forget them!

  • (SignatureMethod)

  • (Transforms)?
  • (DigestMethod)
  • (DigestValue)
  • </Reference>)+
  • </SignedInfo>

Canonicalization Method
  • Refers to how the content is “flattened”
    • Represented as a URI
  • Reason is that you have to be able to reconstruct the exact bytes that were input to the digest algorithm
    • Or else you can’t verify the signature
  • Difficulties arise however…
    • Usual CR/LF stuff...
    • Whitespace…
    • Ordering attributes…
  • C14N is actually a separate XML specification
21
SignatureMethod...
  • Simple enough (phew!):



  • <SignatureMethod Algorithm="http://www.w3.org/2000/09/xmldsig#rsa-sha1" />
22
References…
  • The “business-end” of the signature:
    • Points (using URI) at the signed content
      • URI attribute
    • Tells you which part(s) are signed
      • Transforms element
    • Tells you how the content is digested
      • DigestMethod
    • Contains the digest over the content
      • DigestValue
23
Transforms...
  • Sometimes you only want to sign part(s) of a document (or documents)
    • Other parts might change during later processing
    • Other parts may not be part of some “trust” semantics
  • Transforms element is a very flexible mechanism for doing this
    • Produces either an octet string or an XPath node-set
      • A WHAT? (A bunch of elements, more-or-less)
      • XPath is yet another W3C specification
24
More transforms…
  • Basically a programming language!
  • Given an XML document a transform can turn this into another XML document in almost any way
    • Some transforms are recommended:
    • C14N, base64, XPath, enveloped-signature, XSLT
    • Its even possible to include Java transforms!
  • Transforms can be applied in sequence
    • Output of one is input to next
    • Some octets <-> nodes issues
25
Digests...
  • DigestMethod is an algorithm
    • Algorithm input is output of last transform
    • Algorithm output is DigestValue
  • DigestValue is base64 encoded value
26
SignatureValue
  • Once you’ve assembled the SignedInfo
    • Including the digest(s) etc
  • C14N this..
  • Apply the signing algorithm using the private key
    • Requires digesting again!
  • The SignatureValue contains the output value
    • base64 encoded
  • PKCS#1 v1.5 signing is used
    • So there’s an ASN.1 DigestInfo structure “inside”
27
KeyInfo
  • This optional element contains information to help you verify the signature
  • There are a bunch of variations:
    • KeyValue: an actual public key
    • X509Data: (a bucket of) public key certificates or identifiers (e.g. issuer/serial)
      • Some rules about >1 bucket
    • …others...
28
Object...
  • Basically can contain anything!
  • Some specific uses are described in the specification:
    • Where a reference points at an object
    • Where a “manifest” points into an object
  • Used if the signature envelopes the signed content
29
Manifests…
  • Validation requires you to check the correctness of the digests of all the references
  • Sometimes this isn’t what’s wanted
    • If an application only needs one of the digests to be correct
  • A manifest is like a bunch of references but “core” validation does not require checking each digest
30
SignatureProperties…
  • You may have noticed there’s no reference to signing time or other similar things
  • This is deliberate (remember: no “semantics”)
  • If you do want to include information “about” the signature (as opposed to “about” the content!) then a SignatureProperty element can be included in the Object element and a Reference to this in the SignedInfo
  • Typical uses: timestamps, signing policy, h/w...

Recap signing (1)...
  • Reference Generation, for each data object being signed:
    • Apply the Transforms, as determined by the application, to data object
    • Calculate the digest value over the resulting data object.
    • Create a Reference element, including the (optional) identification of the data object, any (optional) transform elements, the digest algorithm and the DigestValue.
37
Recap signing (2)...
  • Signature Generation:
    • Create SignedInfo element with SignatureMethod,       CanonicalizationMethod and Reference(s).
    • Canonicalize and then calculate the SignatureValue over SignedInfo based on algorithms specified in SignedInfo.
    • Construct the Signature element that includes SignedInfo, Object(s) (if desired, encoding may be different than that used for signing), KeyInfo (if required), and SignatureValue.
38
Validation (1)...
  • Reference Validation, for each Reference in SignedInfo:
    • Canonicalize the SignedInfo element based on the CanonicalizationMethod in SignedInfo.
    • Obtain the data object to be digested. (either based on the identification (URI) and Transforms in the Reference element, or via other means such as a local cache.)
    • Digest the resulting data object using the DigestMethod specified in its Reference specification.
    • Compare the generated digest value against DigestValue in the SignedInfo Reference; if there is any mismatch, validation fails.
39
Validation (2)...
  •  Signature Validation
    • Canonicalize the SignedInfo element based on the CanonicalizationMethod in SignedInfo.
    • Obtain the keying information from KeyInfo or from an external source.
    • Use the specified SignatureMethod to validate the SignatureValue over the SignedInfo element.

Interoperability
  • The specification is good about the format, but requires a lot of expertise to apply
  • There are currently about 10 (known) implementations that claim to interoperate (at least partially)
  • For most people what’ll be interesting are two things:
    • How to specify how signatures are to be used in their application context
    • How to use toolkits to produce & verify signatures
  • Right now we don’t have enough experience of either of these, but getting there...

  • XML encryption (XENC) activity is also pretty much done

Conclusion (sales pitch:-)
  • XMLDSIG and XENC core specifications stable
    • Interoperable implementations exist
  • Next steps are to:
    • Gain experience with using this specification in “open” application contexts (may lead to subtle changes or additions)
    • Continue work to include XMLkey management (XKMS)
  • Meanwhile, toolkits are available and usable for signing in current applications
